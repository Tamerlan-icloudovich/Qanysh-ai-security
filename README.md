# Qanysh-ai-security
The main task at the design stage of the system architecture was the mutual coordination of heavy mathematical calculations (machine learning models) and the smooth operation of the mobile client. Although the option of embedding the model directly inside the smartphone (for offline work) was considered, this decision had to be abandoned. This is because datasets are constantly updated, and the neurodevelopment inside the device uses the smartphone's Ram (RAM) and battery power to the extreme.
In this regard, a microservice (client-server) architecture was chosen for the project. This approach makes the client application as "light" as possible, placing all the analytical burden on backend. The technology stack was formed from the following components:
1) mobile client (Frontend): Android (Kotlin / Java) the Android operating system was chosen as the target platform for managing the user interface and system triggers. Why not iOS? The strict "sandbox" (sandbox) policy of the Apple ecosystem imposes a strict restriction on third-party applications to read incoming SMS messages or intercept links of other applications in the background (intercept). And on the Android platform there are open API tools (Broadcast Receivers, Accessibility Services) that allow you to conduct full-fledged anti-phishing monitoring. This was a critical factor in creating a cybersecurity application.
2) analytical core (machine learning): Python programming language was used to write logic that analyzes the risk level of links and numbers. Although C++ or Java has somewhat lost in terms of processing speed, Python is the absolute standard of data Science and the field of artificial intelligence. Scikit is a rich ecosystem of libraries such as learn, Pandas, and TensorFlow to clean datasets, customize hyperparameters, and the project's main tool is phishing_model.PKL offers an ideal environment for exporting the model.
3) server-side framework (backend API): the modern FastAPI framework has been chosen to act as a bridge between the fastapi user's smartphone and the machine learning model written in Python. There are several architectural reasons for choosing FastAPI instead of the popular Django or Flask solutions in backend development:
1) asynchronous processing (Asynchronous I/O): in the event that the system receives a request to check suspicious links from thousands of users at the same time, FastAPI's async/awake mechanisms ensure that the server is not blocked. It processes requests in parallel, not in turn.
2) high performance: built on the basis of the Starlette and Pydantic libraries, this framework can compete with NodeJS and Go in terms of speed.
3) Data Validation: the format of links in JSON-packets coming from the client is automatically checked by Pydantic. This protects the server from attacks caused by SQL injections or malformed requests.
Integration communication: the overall process is based on the RESTful API architecture. Android-the client sends a link (URL) received from the user to the server using an HTTP POST request in JSON format. FastAPI accepts the request and redirects it to the ML-model in Python. After the model performs a structural analysis, the system calculates the hazard level as a percentage and returns it back to the client within milliseconds. Thus, the principle of effective allocation of computing resources is fully implemented.
Comprehensive cybersecurity systems should not only block threats, but also improve the digital literacy of the user. To achieve this goal, the project architecture implemented the integration of two different intelligent systems — Predictive ML (predictive model) and Generative AI (generative model). This hybrid approach allows the application to identify a threat with mathematical precision and interpret it in human language.
First level: local classification (phishing_model.pkl)
When a suspicious link or number is received from a user through a mobile client, it is first received by a locally trained model on our server. At this stage, the external API (internet) is not used, all calculations take place inside the system. The algorithm disassembles the lexical structure of the link (for example, too many special characters or anomalies in the domain name), calculates the probability that it is phishing on a scale of 0.0 to 1.0.
If the threat level is low (for example, less than 20%), the system immediately returns the signal "safe" to the client, without spending additional resources. And if the probability exceeds the critical threshold (Threshold > 65%), the second level of integration is triggered.
Second level: dynamic consulting via the Gemini API
Traditional applications produce static (hardcoded) text that says "do not follow this link" when a threat is found. In our project, this approach was completely revised. The system creates a suspicious context and sends it to the Google Gemini model via FastAPI.
